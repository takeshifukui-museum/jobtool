# AGENTS.md
## 株式会社Museum｜AI開発チーム共通ルールブック

対象：ChatGPT+Codex / Claude+Claude Code / Gemini（将来）
最終更新：2026-03-31

---

## ■ AGENTS.mdの運用ルール

- プロジェクトフォルダ作成時に必ずAGENTS.mdを配置する
- プロジェクト開始時には必ず最新版のAGENTS.mdを読むこと
- プロジェクト終了後に失敗・学びをもとにレビューを行い、得られた知見をAGENTS.mdに追記する
- 古いバージョンのAGENTS.mdをそのまま使い回すことを禁止する

---

## このファイルを読んだら最初にやること

以下を順番に確認してから作業を開始すること。飛ばし禁止。

1. このAGENTS.mdが最新版であることを確認する（更新日時を確認）
2. 作業対象のリポジトリ名をREADMEで確認する
3. ブランチがmainであることを確認する（git branchで確認）
4. 実装計画を日本語で福井に提示する
5. 福井の承認を得てから実装を開始する

---

## ■ 前提

株式会社Museumの本業は人材紹介業である。
AI活用・開発はすべて人材紹介事業を拡張・加速させるための手段であり、目的ではない。

ツールが完成することが目的ではない。面談数が増えることが目的。

---

## ■ 絶対に守るルール（非交渉制約・違反禁止）

以下は福井の明示的な許可なしに絶対に実行しない。

### Git・GitHubまわり
- ブランチの自動作成禁止。mainブランチのみで作業する
- 新規GitHubリポジトリの作成は明示的な許可なし禁止
- コミット・プッシュは必ず福井の確認を得てから実行する
- プッシュ前に「プッシュしてよいですか？」と確認を取る
- 離席中・応答がない状態での作業・プッシュ禁止
- 作業前に必ずgit remote -vでリポジトリURLを確認する

### 設計・実装まわり
- 推測での設計禁止。不明点は実装前に必ず質問する
- 設計未確定のまま実装に入ることを禁止する
- 一度に大きく作らない。MVP分割・1PR単位・段階的拡張
- 巨大改修は禁止する
- 細部の改善ループに入ることを禁止する（動くものを優先する）

### 自動化まわり
- 完全自動化を原則採用しない
- 送信前停止ポイントを必ず設ける
- CAPTCHA突破禁止
- 手動確認ポイント必須
- スパム行為禁止

### コストまわり
- トークン浪費を避ける
- 並列セッション乱用禁止
- 巨大コンテキスト乱用禁止

## ■ 福井への報告ルール

- 福井が手動で行う操作は必ず【福井・手動対応】と明記する
- 実装前に計画を日本語で説明する
- 完了後に作業内容を日本語でまとめて報告する
- エラーが発生した場合はコードを実際に読んでから原因を特定する（推測で修正しない）
- 福井の指摘・修正事項は最優先で対応する

## ■ 設計の進め方（Spec先行原則）

すべての実装は以下の順で行う。

Spec（何を作るか） → Plan（どう作るか） → Execute（実装）

各仕様には以下を必ず明記する。

- 成功の定義
- 失敗の検知方法
- ログ保存先
- ロールバック方法

## ■ 状態管理原則

すべての自動化は状態遷移を持つ。
機能単位ではなく、状態管理システムとして設計する。

例：NEW → READY → IN_PROGRESS → SENT → NO_REPLY → STOP

ステータスが曖昧な設計は禁止する。

---

## ■ プロジェクト命名規則

すべてのプロジェクトは以下の形式で命名する。

【カテゴリ】_【ツール名】_【年月YYYYMM】

カテゴリ一覧：

- Sales_：営業・開拓系
- Finance_：売上・請求書管理系
- DB_：人材・企業データベース系
- Mkt_：マーケティング系
- Core_：複数ツールを統合した基幹系

設計書作成前に以下の3つを必ず確定する。

1. プロジェクト名（命名規則に従って決定）
2. GitHubリポジトリ名
3. ローカルフォルダ名（C:\dev\プロジェクト名\）

この3つが確定するまで設計書・実装を開始しない。

---

## ■ ファイル命名規則

- 定義ファイル：Claude_ツール名_日付.md（例：Claude_Sales-CompanyFinder_20260326.md）
- batファイル：start_プロジェクト名.bat（例：start_Sales-CompanyFinder.bat）
- 「Claude.md」「起動.bat」という汎用名は禁止

## ■ 開発運用モデル

開発はバックグラウンドで進める。

- 朝：タスク指示（15分以内）
- 昼：営業活動優先
- 昼・夜：確認作業（1〜2時間ごとに10〜20分）

開発が主業になってはならない。

## ■ 過去の失敗・知見（必ず確認すること）

### Git・GitHubまわり
- claude.mdのルールを無視してブランチを自動作成することがあった → 毎回mainブランチを確認してから作業する
- 確認なしでコミット・プッシュを実行することがあった → プッシュ前は必ず確認を取る
- 離席中に勝手に作業を進めることがあった → 福井の応答がない場合は作業を止めて待つ
- 間違ったリポジトリで作業したことがあった → 作業前に必ずgit remote -vでリポジトリURLを確認する
- 存在しないリポジトリを参照し続けることがあった → 作業前にリポジトリの存在を確認する

### 設計まわり
- コードを読まずに推測で改善点を設計し的外れな修正指示を出すことがあった → 必ずコードを実際に読んでから原因を特定する
- 細部の改善ポイントを出し続けて開発が進まないことがあった → 動くものを優先し、細部は後回しにする
- 一度に全機能を実装しようとすることがあった → Phase分割・MVP優先で進める

### 環境まわり
- Windowsのユーザー名に全角スペース（「福井　毅」）が入っているためパスに注意
- batファイルの日本語エンコードがWindowsのcmdと合わずに文字化けすることがあった → batファイルは英語のみ、日本語表示はNode.js側で行う
- PowerShellとClaude Codeの入力欄を混同することがある → 福井への手順説明は【福井・手動対応】と明記する

## ■ 成功の定義

開発成功とは以下をすべて満たすこと。

- 営業時間が減っていない
- 面談数が維持または増加
- 手作業時間が減少
- バグ修正時間が短縮

ツール完成ではない。判定タイミング：開発着手から4週間後。

---

## ■ 緊急停止手順

自動化ツールが意図しない動きをした場合：

1. AIのセッションを即座に閉じる（ブラウザ・アプリを閉じる）
2. git checkout mainで元に戻す
3. 福井に報告する

停止手順は各プロジェクトのREADMEにも必ず記載する。
